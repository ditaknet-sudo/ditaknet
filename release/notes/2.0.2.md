# DitakNet 2.0.2

## Highlights

- Introduces immutable, signed update metadata for future stable and beta
  releases, bound to the exact GHCR multi-architecture image digests.
- Adds hardened Docker and TrueNAS deployment definitions with a non-root
  `568:568` runtime, read-only root filesystem, minimal capabilities, bounded
  processes, persistent storage contracts, and deep health evidence.
- Adds backup-first update preflight and offline-only restore/rollback tooling.
  DitakNet notifies and prepares an update; it never replaces its own Docker
  container or TrueNAS App.

## Changed

- `VERSION` is now the canonical source/build version. The root
  `update-manifest.json` remains the immutable schema-v1 record for the legacy
  `2.0.1` amd64 image.
- Fresh Docker Compose installs use four Docker-managed named volumes. Existing
  bind-mount installations must explicitly preserve all four old paths before
  redeployment.
- Official images are built and smoke-tested independently for `linux/amd64`
  and `linux/arm64`, then published as one exact SemVer multi-architecture
  index.
- Backup archives use format 2 with member checksums, bounded validation,
  SQLite integrity checks, and update/migration operation context.

## Fixed

- Update verification now fails closed for signature, channel, digest,
  compatibility, replay, cache, and network failures.
- A feed failure followed by an application restart no longer re-enables a
  stale cached update handoff during persisted backoff.
- Database startup rejects future schemas, unsafe downgrades, and migration
  fingerprint drift before persistent state can be mutated.
- SQLite backup and restore handling is WAL-safe, bounded, and crash-aware.
- Session-authenticated mutations consistently enforce CSRF protection.
- Update UI content is safely rendered, and release links are restricted to
  approved HTTPS destinations.
- Scheduler retry, TCP refused-connection handling, health reporting, locale
  consistency, and migration idempotency regressions were corrected.

## Security and packaging

- Runtime dependencies and CI dependencies are SHA-256 hash locked.
- Release images are vulnerability-scanned and carry verified provenance plus
  per-platform SPDX SBOM attestations.
- The release workflow refuses to mutate GHCR until the protected channel key
  matches the public key committed in the tagged source.
- Exact release tags are immutable; the workflow never publishes or moves
  `latest`.

## Upgrade from 2.0.1

The published `2.0.1` image predates the signed schema-v2 updater and has no
official update feed configured by default. It cannot automatically discover
or install `2.0.2`. Treat this first transition as a one-time manual bootstrap:

1. preserve the exact Data, Logs, Backups, and Plugins mounts;
2. create and validate a backup, plus a storage snapshot where available;
3. stop the legacy container explicitly;
4. select the exact `2.0.2` image and redeploy with the same mounts;
5. require `/health/deep` to pass before completing the change.

From `2.0.2` onward, the application can automatically check the signed
channel feed and display a trusted update offer. Update application remains an
administrator-controlled external Docker/TrueNAS redeploy after the in-app
backup-first preflight; unattended self-update is intentionally disabled.

Rollback is state-aware. Do not switch only the image tag after `2.0.2` has
written the database. Stop the application and restore the validated
pre-update state using the documented offline process before selecting the old
image.

## Validation scope

- Automated Python, security, release-consistency, backup/restore, update,
  Docker hardening, and TrueNAS packaging tests are required.
- Both `linux/amd64` and `linux/arm64` release images must pass runtime smoke,
  health, persistence, scan, provenance, and SBOM gates in GitHub Actions.
- TrueNAS Compose and catalog variants are statically validated and rendered
  against a pinned official TrueNAS app-library revision. A real TrueNAS SCALE
  install/upgrade/rollback acceptance test remains separate hardware evidence
  and must not be implied by catalog rendering alone.
